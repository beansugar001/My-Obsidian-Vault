### 1. VerilogLAVD: LLM-Aided Rule Generation for Vulnerability Detection in Verilog

- **作者**: Xiang Long, Yingjie Xia, Xiyuan Chen, Li Kuang
- **机构**: Micro-Electronics Research Institute, Hangzhou Dianzi University, Central South University
- **链接**: [arXiv](https://arxiv.org/abs/2508.13092) | [PDF](https://arxiv.org/pdf/2508.13092)
- **来源**: 📄 arXiv 关键词检索
- 📒 **笔记**: [[VerilogLAVD]]

- **核心方法**: 输入是 Verilog RTL 代码 + CWE 漏洞描述，输出是漏洞检测结果。三步走：(1) **VeriPG 构建**：将 Verilog 代码转为统一图表示，融合 AST（语法结构）+ CFG（控制流）+ DDG（数据依赖），通过 Common Node 对齐三种图；(2) **Validation-Based Rule Generation**：LLM 从 CWE 描述生成图遍历检测规则，用遍历原语（Node、Variable、Branch 等）+ 过滤器（DFS/BFS 路径等），同时实时验证规则合法性避免 LLM API 误用；(3) **Rule Executor**：根据规则的 traversal primitives 在 VeriPG 上执行路径匹配检测漏洞。与 [[SecFSM]] 的关键区别：SecFSM 在生成阶段防漏洞，VerilogLAVD 在已有代码上查漏洞。两者都依赖 LLM + 图结构，但目的相反（pre-generation vs post-detection）。
- **对比方法/Baselines**: LLM-only（纯 LLM 检测）、LLM+external knowledge（知识增强 prompt 检测，类似 [[SecFSM]] 中 RAG 的思路）。与 Code Property Graph (CPG) 借鉴关系明确，但针对 Verilog 的并行语义 + 时序特性做了适配。数据集涵盖 77 个 Verilog 设计、12 种 [[CWE]] 类型。
- **借鉴意义**: VeriPG 的统一图表示思路（AST+CFG+DDG fusion）值得注意，对需要做代码结构分析的场景（不仅是 EDA）有参考价值。Validation-Based Rule Generation 的"生成 + 实时校验"循环减少 LLM 幻觉输出，这个模式可以泛化。
- **锐评**: 思路清晰，三个模块（VeriPG → Rule Generation → Rule Executor）各自有明确的设计理由。F1=0.54 不算高但相对 LLM-only（0.23）和 LLM+knowledge（0.27）的 +0.31/+0.27 提升是实质性的。但几个问题：(1) F1=0.54 的绝对分数说明离实用还有距离，precision 和 recall 各是多少？摘要没明确给，需要看正文；(2) 每条规则需要 LLM 生成，77 个设计 12 个 CWE 类型就已经不少了，扩展到上百种 CWE 时 scalability 存疑；(3) VeriPG 构建依赖 AST/CFG/DDG 的工具链质量，对复杂宏、generate 语句、SystemVerilog 特性的支持能力未提及；(4) 与 [[SecFSM]] 同团队但做的是互补问题，两者如何结合（生成时用 SecFSM 防住 + 生成后用 VerilogLAVD 复查）在论文中未讨论。总体而言方法设计合理，但离实际 EDA 流程中的集成还有工程差距。👀
